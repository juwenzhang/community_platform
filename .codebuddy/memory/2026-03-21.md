## 类型错误修复

### vite.config.ts 多实例类型冲突
- **问题**：pnpm 因 peer deps 差异解析出 3 个 vite@6.4.1 实例（jiti@1.21.7 vs jiti@2.6.1, ±@types/node），`@vitejs/plugin-react` 返回的 Plugin 类型与项目 vite 的 Plugin 类型名义不兼容
- **根因**：`vite.config.ts` 不在 `tsconfig.json` 的 `include: ["src/**/*"]` 范围内，IDE 对其使用 inferred project，类型解析路径不可控
- **修复**：为 feed 和 main 应用创建 `tsconfig.node.json`（覆盖 `vite.config.ts`），`tsconfig.json` 通过 `references` 引用
- 新增文件：`apps/feed/tsconfig.node.json`, `apps/main/tsconfig.node.json`
- 修改文件：`apps/feed/tsconfig.json`, `apps/main/tsconfig.json`（添加 references）

### tsconfig.node.json composite 修复
- **问题**：Project References 要求被引用项目必须 `"composite": true` 且不能 `"noEmit": true`
- **修复**：为 feed 和 main 的 `tsconfig.node.json` 添加 `"composite": true`，移除 `"noEmit": true`

### tsconfig 统一继承层级重构
- **动机**：各子项目 tsconfig 大量重复配置，改为根目录维护基础配置 + 子项目 extends
- **新增根配置**：
  - `tsconfig.base.json` — 公共基础（target/module/strict/isolatedModules 等）
  - `tsconfig.app.json` — React 应用（extends base + jsx: react-jsx）
  - `tsconfig.node.json` — Vite 配置文件（extends base + composite: true）
  - `tsconfig.lib.json` — 共享库（extends base + declaration/declarationMap/sourceMap）
- **简化子项目**：
  - `apps/feed/tsconfig.json` → extends `../../tsconfig.app.json`，仅保留 outDir/rootDir
  - `apps/feed/tsconfig.node.json` → extends `../../tsconfig.node.json`，仅保留 include
  - `apps/main/tsconfig.json` → extends `../../tsconfig.app.json`，保留 outDir/rootDir/paths
  - `apps/main/tsconfig.node.json` → extends `../../tsconfig.node.json`，仅保留 include
  - `packages/shared-types/tsconfig.json` → extends `../../tsconfig.lib.json`，保留 jsx/outDir/rootDir
- **注意**：`outDir`/`rootDir` 等路径相关选项不能放在根配置里（它们相对于配置文件位置解析），必须由子项目各自指定

## 修复 tsc -b 产生中间编译产物 (vite.config.d.ts / vite.config.js / .tsbuildinfo)

### 问题
- `tsc -b` (project references 模式) 会将 `vite.config.ts` 编译输出为 `vite.config.d.ts` + `vite.config.js` 到源文件同级目录
- 同时产生 `tsconfig.tsbuildinfo` / `tsconfig.node.tsbuildinfo` 增量编译缓存文件
- 这些中间产物污染了项目目录

### 根因
- `tsc -b` 是「编译模式」，会真正输出文件；`tsconfig.node.json` 中 `composite: true` + `include: ["vite.config.ts"]` 导致 vite 配置文件被编译成 .js/.d.ts
- 而在 Vite 项目中，tsc 只需要做类型检查，实际构建由 Vite 负责

### 修复
- `tsc -b` → `tsc --noEmit`（只检查类型，不输出任何文件）
- 去掉 `tsconfig.json` 中的 `composite`、`outDir`、`references`（`--noEmit` 不需要这些）
- `.gitignore` 追加 `vite.config.d.ts` / `vite.config.js` 作为额外保险

### 文件变更
- `apps/feed/package.json` — build script: `tsc -b` → `tsc --noEmit`
- `apps/feed/tsconfig.json` — 去掉 composite/outDir/references
- `apps/main/package.json` — build script: `tsc -b` → `tsc --noEmit`
- `apps/main/tsconfig.json` — 去掉 outDir/references
- `.gitignore` — 追加 vite.config.d.ts / vite.config.js 忽略规则

## 修复 preview 模式子应用加载失败（第二轮）

### 上轮遗留问题
前一轮修复后 preview 仍然 `Failed to fetch`，因为真正的根因未解决。

### 深层根因
`EnvConfigProvider` 中的 `getEnvMode()` 使用间接方式访问 `import.meta.env`：
```ts
const meta = import.meta as unknown as { env?: { MODE?: string } };
return meta.env?.MODE ?? 'development';
```
通过 `as unknown as` 类型转换后，Vite 的静态分析**无法识别**这是 `import.meta.env.MODE` 的访问，
因此**没有将其替换为 `"production"` 字符串**。

构建产物中保留了 `import.meta.env` 的运行时访问，
在浏览器中 `import.meta.env` 是 `undefined`（仅 Vite dev server 注入），
所以 `getEnvMode()` 始终 fallback 到 `"development"`，
`EnvConfigProvider` 选了 `envEntries.development` = `"http://localhost:5174"` → 连不上。

### 修复方案
1. **修改 `getEnvMode()`**：改用 `import.meta.env.MODE` 直接语法（Vite 可静态替换）
2. **新增 `env.d.ts`**：为 `app-registry` 包添加 `ImportMeta` 类型声明（包不依赖 vite）

### 验证结果
- 构建产物中 `import.meta.env` 出现 0 次（之前 1 次）✅
- `getEnvMode()` → 直接返回 `"production"` ✅
- Garfish 正确从 `/apps/feed/` 加载子应用 ✅

### 文件变更
- `packages/app-registry/src/providers/env.ts` — 修复 getEnvMode 使用直接访问语法
- `packages/app-registry/src/env.d.ts` — 新增 ImportMeta 类型声明
